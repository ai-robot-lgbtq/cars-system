## 1. 项目概述

### 1.1 项目背景

本项目是一个面向学习与实践的**全闭环二手车交易平台**。在学习 React + TypeScript + Go 全栈开发的同时，落地一个具备真实业务模型的系统，覆盖发布、交易、支付、聊天、评价、售后等核心流程。

### 1.2 项目目标

| 维度 | 目标 |
|---|---|
| **技术深度** | 完整跑通 React+TS+Go 全栈开发链路，掌握 Gin、GORM、Next.js App Router、WebSocket、asynq、PostgreSQL 全套技术 |
| **工程实践** | 落地 RESTful API 设计、JWT 鉴权、RBAC、订单状态机、接口抽象、依赖注入、Docker 编排、CI 等工程实践 |
| **业务理解** | 完整经历二手交易核心业务流，理解订单状态、超时关单、支付回调等关键场景 |
| **代码质量** | 关键路径有单元测试 + 集成测试，关键模块有 E2E 测试覆盖 |

### 1.3 系统范围

**包含：**

- ✅ 4 种角色：游客、买家、卖家、管理员
- ✅ 5 种登录方式：邮箱+密码、手机号+验证码、微信 OAuth、支付宝 OAuth（仅登录）
- ✅ 车辆发布、浏览、搜索、收藏、审核
- ✅ 订单全状态机：创建 → 支付 → 发货 → 过户 → 完成
- ✅ 本地 Mock 支付（预留微信支付 V3 接口，便于后续接入）
- ✅ WebSocket 实时聊天 + 站内消息中心
- ✅ 双向评价、售后工单
- ✅ 独立管理端：审核、封号、数据看板
- ✅ Docker Compose 一键启动

**不包含（Non-Goals）：**

- ❌ 真实物流对接（仅平台内流程跟踪）
- ❌ 真实支付（仅 Mock，结构预留接口）
- ❌ 短信服务真实对接（仅接口预留，可注入 Mock）
- ❌ OAuth 真实账号审核流程（沙箱账号或 Mock 回调）
- ❌ 移动端 App（仅响应式 Web）
- ❌ Kubernetes 部署
- ❌ 微服务化拆分（单进程 Go 后端 + 单 asynq worker）

### 3.2 关键设计原则

| 原则 | 说明 |
|---|---|
| **前后端分离** | 两个 Next.js 应用 + 一个 Go 后端，独立部署 |
| **后台独立** | 管理端与用户端代码可共享，但部署独立（不同端口） |
| **接口抽象** | WebSocket Broker、Payment Gateway 等关键能力以接口抽象，环境切换不影响业务 |
| **职责分层** | handler（HTTP/WS 边界）→ service（业务逻辑）→ repository（数据访问）三层 |
| **业务不依赖框架** | service 层不引用 gin/gorm 类型，便于测试与替换 |

### 3.3 WebSocket Hub：双实现

抽象接口：

```go
type Broker interface {
    Publish(channel string, msg []byte) error
    Subscribe(channels []string, handler func(channel string, msg []byte)) error
}
```

| 环境 | 实现 | 说明 |
|---|---|---|
| 本地/测试 | `MemoryBroker` | 单进程内 channel map，零依赖 |
| 生产（预留） | `RedisBroker` | Redis Pub/Sub，不增加新组件（Redis 已被 asynq 使用） |

切换：`WS_BROKER=memory|redis` 环境变量，业务代码无需改动。

## 4. 模块设计

### 4.1 后端模块清单

每个模块遵循 **handler → service → repository** 三层结构。

| 模块 | 职责 | 关键接口 |
|---|---|---|
| **auth** | 注册/登录/验证码/OAuth 回调/JWT 签发与刷新 | `AuthService.Signup/Login/Refresh/HandleOAuthCallback` |
| **user** | 用户资料、实名信息、地址簿 | `UserService.GetProfile/UpdateProfile` |
| **catalog** | 车辆发布、上下架、审核、图片上传、浏览/搜索 | `CarService.Publish/List/Search/Online/Offline` |
| **order** | 下单、状态机、超时关单、退款触发 | `OrderService.Create/Pay/Ship/Transfer/Confirm/Cancel` |
| **payment** | 支付网关抽象 + 本地 Mock 实现 | `PaymentGateway.Create/HandleNotify/Query/Refund` |
| **chat** | WebSocket Hub、消息持久化、已读未读 | `ChatService.Send/History/MarkRead` |
| **notification** | 站内消息中心（订单/系统/聊天消息聚合） | `NotificationService.List/MarkRead` |
| **review** | 双向评价、评分聚合 | `ReviewService.Submit/ForUser/Aggregate` |
| **aftersales** | 售后工单（仅退款/退货退款） | `AfterSalesService.Create/Process` |
| **admin** | 审核、封号、数据看板 | `AdminService.AuditCar/BanUser/Stats` |
| **upload** | 图片上传、签名校验、MIME/大小校验 | `UploadService.UploadImage` |
| **shared** | 中间件、错误码、配置加载、迁移、RBAC | — |

### 4.2 三层结构示例

```
internal/order/
├── handler.go          # HTTP Handler（绑定路由、参数解析、响应格式化）
├── service.go          # 业务逻辑（状态机转换、调用 repository）
├── repository.go       # 数据访问（封装 GORM 调用）
├── model.go            # 领域模型（Order 实体、OrderStatus 枚举）
└── service_test.go     # service 层单元测试（mock repository）
```

| 任务 | 触发时机 | 失败重试 |

### 5.3 关键索引

```sql
-- 车辆浏览/搜索
CREATE INDEX idx_cars_status_created ON cars (status, created_at DESC);
CREATE INDEX idx_cars_brand_model ON cars (brand, model);
CREATE INDEX idx_cars_price ON cars (price);
CREATE INDEX idx_cars_seller ON cars (seller_id);

-- 全文搜索（中文）
CREATE EXTENSION IF NOT EXISTS zhparser;
CREATE INDEX idx_cars_search ON cars USING GIN (
    to_tsvector('zhparser', coalesce(brand,'') || ' ' || coalesce(model,'') || ' ' || coalesce(description,''))
);

-- 订单
CREATE INDEX idx_orders_buyer_status ON orders (buyer_id, status);
CREATE INDEX idx_orders_seller_status ON orders (seller_id, status);
CREATE INDEX idx_orders_status_created ON orders (status, created_at DESC);

-- 消息
CREATE INDEX idx_messages_conv_time ON messages (conversation_id, created_at DESC);
CREATE INDEX idx_conversations_buyer ON conversations (buyer_id, last_message_at DESC);
CREATE INDEX idx_conversations_seller ON conversations (seller_id, last_message_at DESC);

-- 通知
CREATE INDEX idx_notifications_user_unread ON notifications (user_id, read_at)
    WHERE read_at IS NULL;
```

### 5.4 关键设计决策

| 决策 | 选择 | 原因 |
|---|---|---|
| 主键 | `BIGSERIAL` | PG 自增、Join 性能好；不需要雪花算法的全局唯一 |
| 金额 | `NUMERIC(12,2)` | 严禁 FLOAT，避免精度问题 |
| 软删 | `deleted_at`（仅 users/cars/orders） | 业务数据需追溯 |
| 时间 | `TIMESTAMPTZ` + UTC | 国际化友好 |
| 实名信息 | 加密字段（AES） | 合规基础 |
| JSONB | audit_logs / payment_logs / configs | 灵活扩展 |
| 角色 | SMALLINT 位运算（1=buyer, 2=seller, 4=admin） | 未来用户多角色预留 |
| 中文分词 | zhparser | 与 PG 集成，学习成本低 |

## 6. API 设计

### 6.1 通用约定

| 项 | 规范 |
|---|---|
| Base URL | `/api/v1` |
| 认证 | `Authorization: Bearer <jwt>` |
| Token 生命周期 | access 15 分钟 + refresh 7 天 |
| 响应格式 | `{ code: 0, message: "ok", data: {...} }` |
| 成功标识 | `code === 0` |
| 分页 | `?page=1&page_size=20`，响应 `data: { items, total, page, page_size }` |
| 时间格式 | ISO 8601（`2026-07-02T10:30:00Z`） |
| API 版本 | URL 路径版本（`/api/v1`），后续 v2 共存 |
| 角色访问 | 游客可读 `GET /api/v1/cars*`、`GET /api/v1/users/:id`、`GET /api/v1/users/:id/reviews`；其他接口需登录；`/seller/*` 需 seller 角色；`/admin/*` 需 admin 角色 |

## 9. 质量保障

### 9.1 测试策略

| 层级 | 工具 | 覆盖目标 |
|---|---|---|
| 单元测试 | `testing` + `testify` | service 层关键路径 ≥ 70% 覆盖率（业务规则、状态机转换） |
| 集成测试 | `dockertest` + GORM | repository 层 + 关键 API（auth/订单）端到端 |
| 前端单测 | Vitest + RTL | 关键组件、hooks |
| E2E | Playwright | 核心用户路径：注册→发布→下单→支付→聊天→评价 |

### 9.2 Lint 与格式化

| 语言 | 工具 | 用途 |
|---|---|---|
| Go | golangci-lint | 综合 lint |
| Go | gofmt / goimports | 格式化 |
| TS/TSX | ESLint + Prettier | 规范与格式化 |

### 9.3 CI 流程（GitHub Actions）

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  backend:
    - checkout
    - setup-go 1.22
    - go mod download
    - golangci-lint run
    - go test ./... -race -coverprofile=cover.out
  frontend:
    - checkout
    - setup-node 20
    - pnpm install
    - pnpm lint
    - pnpm typecheck
    - pnpm test
    - pnpm build
```

### 9.4 错误处理约定

- **业务错误**：使用统一错误码（如 `30001 = 库存不足`、`30100 = 订单状态不允许此操作`）
- **Panic 兜底**：Gin `Recovery` 中间件 + zap 记录堆栈
- **降级策略**：核心接口必须有兜底（如搜索降级到 LIKE 查询）

### 9.5 关键安全点

- 密码 bcrypt（cost=12）
- JWT 密钥从环境变量注入，不入库
- 实名信息 AES 加密
- SQL 注入：全程 GORM 参数化，零字符串拼接
- XSS：前后端双重转义；Ant Design 默认转义
- CSRF：Bearer Token + SameSite Cookie
- 上传文件：MIME 嗅探 + 扩展名校验 + 大小限制 + 随机文件名
